# Windows setup

This guide uses portable paths and placeholders. The default installer location is:

```text
%LOCALAPPDATA%\EmbodiedArtificialLife
```

## Prerequisites

- Windows 10 or 11
- `uv` available on `PATH`
- Python 3.11, which `uv` can install automatically
- Optional: LM Studio or another OpenAI-compatible local model server
- Optional: Tailscale for private remote access

## Install or repair the latest release

Open PowerShell:

```powershell
$installer = "$env:TEMP\install-embodied-alife.ps1"
Invoke-WebRequest `
  -Uri "https://raw.githubusercontent.com/eWOOD29/embodied-alife/main/install-windows.ps1" `
  -OutFile $installer
PowerShell -ExecutionPolicy Bypass -File $installer
```

To choose a different location:

```powershell
PowerShell -ExecutionPolicy Bypass -File $installer `
  -InstallPath "D:\Apps\EmbodiedArtificialLife"
```

Running the installer again is the supported repair path when an older installation cannot update itself. It preserves `.env`, `.venv/`, `data/`, and `.git/`.

## Run locally

Double-click:

```text
start-embodied-alife.bat
```

Or use PowerShell:

```powershell
$root = "$env:LOCALAPPDATA\EmbodiedArtificialLife"
Set-Location $root
& .\.venv\Scripts\python.exe -m app.serve
```

Use `app.serve`, not raw `uvicorn app.main:app`, because the managed launcher provides updater shutdown coordination.

Default endpoints:

```text
Dashboard:       http://127.0.0.1:8797/
Health:          http://127.0.0.1:8797/health
Soak readiness: http://127.0.0.1:8797/api/validation/readiness
```

## AppDock

The tracked `appdock.json` is portable. Its `projectDirectory` is `.` and must be resolved relative to the directory containing `appdock.json`; it launches the project-local `.venv\Scripts\python.exe` with `-m app.serve`.

Import or register the manifest from the installed application directory. Do not edit the tracked manifest to contain a username, home-directory path, hostname, Tailnet name, or workspace path. An AppDock build that cannot resolve a manifest-relative project directory should use an untracked local AppDock registration or override generated by AppDock itself, while leaving the packaged manifest unchanged.

## Configure a local model

LM Studio commonly exposes its OpenAI-compatible API at:

```text
http://127.0.0.1:1234/v1
```

1. Load a model in the local model server.
2. Start the OpenAI-compatible API.
3. Open **Local LLM brain** in the dashboard.
4. Click **Discover models**.
5. Select the exact loaded model ID.
6. Enable local LLM use and click **Save and apply**.

Runtime settings persist at `data\runtime\llm-settings.json`. Do not commit that file or include it in public issue attachments without checking it for secrets.

## Private Tailscale access

The application has no independent login. Keep it on localhost or a trusted private network.

Tailscale Serve example:

```powershell
tailscale serve --bg --https=8797 http://127.0.0.1:8797
tailscale serve status
```

Open the HTTPS hostname shown by `tailscale serve status`. Do not publish that hostname, use Tailscale Funnel, or configure public port forwarding.

## In-app updates

The dashboard automatically checks stable GitHub Releases. When an update is available:

1. Review the formatted release summary.
2. Click **Install v…**.
3. Confirm the installation.
4. Allow the page to reconnect after restart.

Updater files are stored under:

```text
data\runtime\update-worker.log
data\runtime\update-state.json
data\runtime\update-backups\
data\runtime\updates\
```

The local Ari provenance key and proof ledger are also stored under the preserved runtime root:

```text
data\runtime\ari-provenance.key
data\runtime\ari-memory-proofs.json
```

These files contain no packaged secret and are created locally. Back them up with `data/`. If an existing state contains provenance proofs but its key is missing or damaged, Ari-facing authorization fails closed; the application does not silently re-sign unverified records.

## Development checkout

From a source checkout:

```powershell
uv venv --python 3.11
uv pip install --python .venv\Scripts\python.exe -e ".[dev]"
Copy-Item .env.example .env
```

Verification:

```powershell
.\.venv\Scripts\python.exe -m pytest -q
.\.venv\Scripts\python.exe -m compileall -q app tests scripts
.\.venv\Scripts\python.exe scripts\validate_package.py
.\.venv\Scripts\ruff.exe check app tests scripts
```

## Back up persistent state

```powershell
$root = "$env:LOCALAPPDATA\EmbodiedArtificialLife"
$stamp = Get-Date -Format "yyyyMMdd-HHmmss"
$destination = Join-Path $env:USERPROFILE "Desktop\embodied-alife-data-$stamp"
Copy-Item "$root\data" $destination -Recurse
Write-Host "Backup created at $destination"
```

Never delete `data/`, `.env`, or `.venv/` as a generic troubleshooting step without understanding the consequences.
